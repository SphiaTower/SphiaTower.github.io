安卓中使用Singleton的问题

Posted on 2015-09-28

在http://sphiatower.github.io/2015/09/23/wrong-use-of-application/中提到了，安卓官方文档中，不建议使用Application子类来支持全局访问。而应使用静态单例，但是在静态单例的使用中也有一些问题。标准的使用方法是这样的：

private Singleton sInstance;
public Singleton getInstance(Context context){
    if(sInstance == null) {
        sInstance = new Singleton(context.getApplicationContext());
    }
    retrun sInstance:
}

然后通过调用Singleton.getInstance(Context)就可以在全局范围内调用Singleton了。

但这样做的一个副作用就是，每次调用的时候都需要传递一个Context进去。当然，理论上将，这些调用的场景，本身就需要Context来发挥作用，所以无可厚非。但实际应用时有两点问题：

  1. 一些类在设计时没有选择获取或储存Context，但又要用到便利的全局方法（比如显示一个Toast）。
  2. 传入的Context仅仅在第一次创建的时候才起作用，其他情况下并无实际作用。

第一个问题，固然可以更改类的设计来传送Context，但太过麻烦也没有必要。第二个问题，归根结底是单例的’Lazy Initialization’提供的便利所带来的代价。

所以，对一些常用的开销小的工具类而言，不妨在应用的一开始就生成，并获取Context，在之后的调用中直接获取单例，不再传送Context。也就是提供一个注册方法，在Application.onCreate()或程序的其他入口处执行。

public Singleton getInstance(){
    return sInstance;
}

public void register(Context context){
    sInstance = new Singleton(context.getApplicationContext());
}

那么，针对开销较大的单例，比如图片缓存的LruCache，怎么避免不必要的创建呢？

方法是使用FlyWeight设计模式。仅仅在创建时获取Context，在调用时再决定是否生成LruCache。

private Singleton(Context context){
    mCtx = context;
}

public Singleton getInstance(){
    if(mCache == null){
        // create the LruCache
        mCache = ...;
    }
    return sInstance;
}
