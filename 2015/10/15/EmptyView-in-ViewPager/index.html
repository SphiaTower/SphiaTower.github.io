EmptyView in ViewPager

Posted on 2015-10-15

Today I encountered a very subtle bug that the Fragments restored from a ViewPager show its empty view with their data already loaded.

My CustomFragment shows an empty view when initialized, and switch to the RecyclerView once data is loaded. The process of animation has been summarized in one of my old articles(http://sphiatower.github.io/2015/09/08/viewpager/). In short, it hides the RecyclerView and shows the empty view in the onCreateView callback.

Every time the data of RecyclerView is updated, it will check an empty flag whether it’s the first time of loading data. If true, the cross fade animation is played to show the content and the flag is updated to remember that data has been loaded.

It works well as long as not in a ViewPager. When the fragment is created, it shows the empty view and waits for data to supplement. At the first time data is updated, the empty flag is true, the cross face animation is played, and then empty is set false. So the next time data is updated, it won’t play the animation. When through a configuration change, the old CustomFragment instance is detached and destroyed. The new CustomFragment will experience the same process as the old one. So the empty view will be displayed for a short moment before it fades out, if data restoration works fine.

However, the behavior becomes strange in a ViewPager. When a fragment is switched away for a distance exceeding the offscreenPageLimit of ViewPager, and then switched back, it shows the empty view instead of the content. I did a lot of experiments and finally figured it out.

Fragments beyond offscreenLimit are not destroyed. I used a FragmentPagerAdapter to setup the ViewPager, and call setOffscreenPageLimit(2). When the fragment is away from the current page and the distance exceeds offscreenLimit, onDestroyView() is called on it, however, onDestroy() is not.
It’s still the same instance of CustomFragment class when you switch back. And when switched back, the fragment calls life cycle callbacks, onCreatView(), onViewCreated() and onActivityCreated(), while onAttach() and onCreate() are not called.

So in my case, the empty flag indicating whether it’s still empty remains false, because the fragment is still the same instance. Once the data is loaded again, onCreateView() is called again to initialize the view, however, the cross fade animation will never be played because of the false value of empty flag.

Solving this problem is not difficult. If the flag is in the scope of CustomFragment, just set it true at the start of onCreateView(), or force to play a cross face animation if the flag is false. If the flag is in a different relating class and you can’t access it in onCreateView(), you should add another flag to record whether the cross fade animation has been played. If true, it indicates that the view is destroyed while the data still exists and external flag regards it as not empty, so you should start the animation by yourself.
