Simulate WPF Data Binding in Android, Part III - Support More Types

Posted on 2016-05-19

In the last two posts, we succeed simulating the WPF data binding for the String property in Android. However, it is obviously not enough. TextView and EditText not only display Strings, but also Integers, Doubles and even more complicated types. Now we try to support more types for our binder.

Get Property Type

If we just support String property, it’s easy to declare every type String. But now we have no idea on the type of the property. Do we need to make the binder class generic and pass a Class<T> argument due to the type erasure? The answer is no.

We could easily get the type of property at runtime by reflex. We find the getter first, and then obtain its return type by reflex.


And with the Class<?> instance, we could find the setter easily. Once we get the getter and setter, we could encapsulate the accession of property in a custom class PropertyHandler.

Parse View Content

It’s not difficult to display different types in a TextView. You just need to call the toString() method whatever the type is. However, parsing the content of TextView into a property needs additional attention. We should switch on the type of the property by instanceof branches, and write different code for different types.


The code above supports Integer, Double, Long and String types in java. If we write it in java code, it should be Integer.parseInt() and so on. However, there’s a severe problem unhandled. If the input of user could not be cast to the type of the property, it will crash the app. So we modify it to make it safe for user inputs.

In Kotlin, checked exceptions are not encouraged. We use null to imply the failure of the parse process. And we simply ignore null results to avoid crash when updating a property from an input.
