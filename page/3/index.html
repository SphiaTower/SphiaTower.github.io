使用Preferences API（二）

Posted on 2016-02-05

使用pref header

某些情况下，可能需要设计设置页面来使第一个屏幕展示一列子屏幕。在3.0以上版本中，应该使用新的header特性，而非使用嵌套的PreferenceScreen元素。

  1. 分离各组设置至不同的PF实例。每组设置需要单独的XML文件。
  2. 创建XML header文件，列出每个设置组，声明含有对应设置列表的fragment。
  3. 继承PA类来持有设置。
  4. 实现onBuildHeaderes()回调来指定header文件。

这样做的一个好处是，PA可以自动在大屏幕中使用双面板布局呈现设定。

使用Preferences API（一）

Posted on 2016-02-05

应用一般都包含一些用户可以自定义的设置，比如是否推送通知，更新数据的频率等等。

如果想要为应用提供设置，应该使用安卓的PreferenceAPI来创建一致的用户体验。

概览

与使用View创造界面不同，设置是由xml文件中声明的各种Preference的子类组成。

一个Preference对象构成一个设置。每个Preference显示为列表中的一个子项，并提供合适的UI供用户交互。例如，CheckBoxPreference创造一个勾选框子项，ListPreference创造一个弹出选择列表对话框的子项。

每个Preference在系统使用的默认的SharedPreferences文件中存在一个对应的键值对。用户更改设置后，系统在SP文件中更新相应的值。只有当需要从SP中读取值以决定应用行为时，才需要直接与SP相交互。

A Bug of TabLayout

Posted on 2016-01-04

In Google’s design support library, there is a bug for version 22.2.1 that the TabLayout in Toolbar may not display. I spent neary an hour debugging for this problem. There is a work-around for this bug:

tabLayout.post(new Runnable() {
    @Override
    public void run() {
        tabLayout.setupWithViewPager(viewPager);
    }
});

But the easiest way is to upgrade the design library to version 23.

安卓导航 - 提供合适的后退导航

Posted on 2016-01-03

后退导航是指用户如何从访问过的屏幕中向后移动。所有的安卓设备都提供有一个后退键，所以你的应用不应该再增加一个后退按钮。

在大多数的情况中，系统在用户导航时维持着一个后退栈。这使得系统能够再用户按下后退键之后合适地向后导航。然而，有一些情况下，你需要手动指明后退行为，以提供更好的用户体验。

需要手动指明后退行为的导航模式有：

  • 用户通过一条通知、一个桌面挂件或导航抽屉直接进入到一个深层次的activity时。
  • 用户在fragment见导航的某些情况。
  • 用户在WebView的网页中导航时。

安卓导航 - 提供Up导航

Posted on 2016-01-03

所有不是主入口的屏幕都应该为用户提供一个通过action bar的Up按钮导航至逻辑父屏幕的方式。

明确父Activity

要实现Up导航，第一步是为每个activity声明合适的父activity。这样系统就可以在manifest中获得activity的逻辑父页面，方便实现导航模式。

从安卓4.1开始，可以通过在<activity>中指明android:parentActivityName属性来声明逻辑父activity。 4.0以下的版本这里略去。

A Glide Sizing Problem

Posted on 2016-01-03

Months ago, I encountered a very frustrating bug of Glide. I was using Glide to display images in a RecyclerView. The image was wrapped in a custom layout, and it was added dynamically. No matter how I adjust the code, images can’t be displayed with its width matching parent. Sometimes it was even scaled into a pixel point!

Until today, I found that the problem was because that Glide figured out the wrong dimensions, as is described here - “https://github.com/bumptech/glide/wiki/Transformations“.

I’m not sure about why Glide get the wrong dimension. But the solution is pretty easy: just call .override(int,int) to set the dimension by yourself. And a .fitCenter() call can then make sure that the scaling of width and height is not changed.

LoaderManager Compares References

Posted on 2015-12-20

Today, one of my Loader implementations behaved totally weird after I modified its code. The Loader loaded successfully at the 1st time, but failed on all following attempts. The loadInBackground() method returned, but onLoadFinished() was not called.

The LoaderManager class compares the new loaded data and old retained data by code like if(oldData!=newData) to decide whether to update the data or not. Note that it uses == operator to compare them. In my case, my Loader retains a same TreeMap object and add/update entries into it, resulting that they’re identified as the same reference, so the loader won’t deliver the new result.

To solve this problem, you have to create a new TreeMap and add back all the old data.

Jcenter peer not authenticated

Posted on 2015-12-16

Today my gradle failed to sync because the following error:

Error: peer not authenticated

I found lots of solutions but none of them take effects, only to know it is caused by the problem of certificate of jCenter().

Finally, it was fixed by replacing all the jCenter() into mavenCentral() in all build.gradle files. Surprisingly, the local gradle repos work without modification.
